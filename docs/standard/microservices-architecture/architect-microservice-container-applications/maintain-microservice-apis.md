# <a name="creating-evolving-and-versioning-microservice-apis-and-contracts"></a><span data-ttu-id="d907e-104">Creación, desarrollo y control de versiones de los contratos y las API de microservicio</span><span class="sxs-lookup"><span data-stu-id="d907e-104">Creating, evolving, and versioning microservice APIs and contracts</span></span>

<span data-ttu-id="d907e-105">Una API de microservicio es un contrato entre el servicio y sus clientes.</span><span class="sxs-lookup"><span data-stu-id="d907e-105">A microservice API is a contract between the service and its clients.</span></span> <span data-ttu-id="d907e-106">Solo podrá desarrollar un microservicio de forma independiente si no incumple el contrato de su API. Este es el motivo por el que el contrato es tan importante.</span><span class="sxs-lookup"><span data-stu-id="d907e-106">You will be able to evolve a microservice independently only if you do not break its API contract, which is why the contract is so important.</span></span> <span data-ttu-id="d907e-107">Cualquier cambio en el contrato afectará a sus aplicaciones cliente o a la puerta de enlace de API.</span><span class="sxs-lookup"><span data-stu-id="d907e-107">If you change the contract, it will impact your client applications or your API Gateway.</span></span>

<span data-ttu-id="d907e-108">La naturaleza de la definición de API depende del protocolo que esté utilizando.</span><span class="sxs-lookup"><span data-stu-id="d907e-108">The nature of the API definition depends on which protocol you are using.</span></span> <span data-ttu-id="d907e-109">Por ejemplo, si usa mensajería (como [AMQP](https://www.amqp.org/)), la API consiste en los tipos de mensaje.</span><span class="sxs-lookup"><span data-stu-id="d907e-109">For instance, if you are using messaging (like [AMQP](https://www.amqp.org/)), the API consists of the message types.</span></span> <span data-ttu-id="d907e-110">Si usa servicios HTTP y RESTful, la API consiste en las direcciones URL y los formatos JSON de solicitud y respuesta.</span><span class="sxs-lookup"><span data-stu-id="d907e-110">If you are using HTTP and RESTful services, the API consists of the URLs and the request and response JSON formats.</span></span>

<span data-ttu-id="d907e-111">Pero, aunque piense en su contrato inicial, una API de servicio debe cambiar con el tiempo.</span><span class="sxs-lookup"><span data-stu-id="d907e-111">However, even if you are thoughtful about your initial contract, a service API will need to change over time.</span></span> <span data-ttu-id="d907e-112">Normalmente, cuando esto ocurre, y especialmente si su API es una API pública utilizada por varias aplicaciones cliente, no puede forzar a todos los clientes a actualizar la versión a su nuevo contrato de API.</span><span class="sxs-lookup"><span data-stu-id="d907e-112">When that happens—and especially if your API is a public API consumed by multiple client applications—you typically cannot force all clients to upgrade to your new API contract.</span></span> <span data-ttu-id="d907e-113">Lo más habitual es implementar progresivamente nuevas versiones de un servicio, de forma que se ejecuten simultáneamente las versiones anteriores y las nuevas de un contrato de servicio.</span><span class="sxs-lookup"><span data-stu-id="d907e-113">You usually need to incrementally deploy new versions of a service in a way that both old and new versions of a service contract are running simultaneously.</span></span> <span data-ttu-id="d907e-114">Por lo tanto, es importante contar con una estrategia para el control de versiones del servicio.</span><span class="sxs-lookup"><span data-stu-id="d907e-114">Therefore, it is important to have a strategy for your service versioning.</span></span>

<span data-ttu-id="d907e-115">Cuando los cambios en la API son pequeños, por ejemplo, si agrega atributos o parámetros a la API, los clientes que usen una API anterior deberán cambiar y trabajar con la nueva versión del servicio.</span><span class="sxs-lookup"><span data-stu-id="d907e-115">When the API changes are small, like if you add attributes or parameters to your API, clients that use an older API should switch and work with the new version of the service.</span></span> <span data-ttu-id="d907e-116">Usted puede proporcionar los valores predeterminados para los atributos que falten y que sean necesarios, y los clientes pueden pasar por alto cualquier atributo de respuesta adicional.</span><span class="sxs-lookup"><span data-stu-id="d907e-116">You might be able to provide default values for any missing attributes that are required, and the clients might be able to ignore any extra response attributes.</span></span>

<span data-ttu-id="d907e-117">Pero en ciertas ocasiones necesitará realizar cambios importantes e incompatibles en una API de servicio.</span><span class="sxs-lookup"><span data-stu-id="d907e-117">However, sometimes you need to make major and incompatible changes to a service API.</span></span> <span data-ttu-id="d907e-118">Puesto que es posible que no pueda forzar a los servicios o aplicaciones cliente a que se actualicen inmediatamente a la nueva versión, un servicio debe admitir versiones anteriores de la API durante cierto período de tiempo.</span><span class="sxs-lookup"><span data-stu-id="d907e-118">Because you might not be able to force client applications or services to upgrade immediately to the new version, a service must support older versions of the API for some period.</span></span> <span data-ttu-id="d907e-119">Si está utilizando un mecanismo basado en HTTP, como REST, una opción es insertar el número de versión de la API en la dirección URL o en un encabezado HTTP.</span><span class="sxs-lookup"><span data-stu-id="d907e-119">If you are using an HTTP-based mechanism such as REST, one approach is to embed the API version number in the URL or into a HTTP header.</span></span> <span data-ttu-id="d907e-120">A continuación, puede decidir si quiere implementar ambas versiones del servicio al mismo tiempo en la misma instancia de servicio o si prefiere implementar distintas instancias y que cada una controle una versión de la API.</span><span class="sxs-lookup"><span data-stu-id="d907e-120">Then you can decide between implementing both versions of the service simultaneously within the same service instance, or deploying different instances that each handle a version of the API.</span></span> <span data-ttu-id="d907e-121">Una buena opción es utilizar el [patrón mediador](https://en.wikipedia.org/wiki/Mediator_pattern) (por ejemplo, la [biblioteca MediatR](https://github.com/jbogard/MediatR)) para desacoplar las diferentes versiones de implementación en los controladores independientes.</span><span class="sxs-lookup"><span data-stu-id="d907e-121">A good approach for this is the [Mediator pattern](https://en.wikipedia.org/wiki/Mediator_pattern) (for example, [MediatR library](https://github.com/jbogard/MediatR)) to decouple the different implementation versions into independent handlers.</span></span>

<span data-ttu-id="d907e-122">Por último, si utiliza una arquitectura REST, [Hypermedia](https://www.infoq.com/articles/mark-baker-hypermedia) es la mejor solución para controlar las versiones de los servicios y permitir las API avanzadas.</span><span class="sxs-lookup"><span data-stu-id="d907e-122">Finally, if you are using a REST architecture, [Hypermedia](https://www.infoq.com/articles/mark-baker-hypermedia) is the best solution for versioning your services and allowing evolvable APIs.</span></span>
